致命问题 (CRITICAL) — 必须立即修复

致命 逻辑错误 backend/api/chat.py:586-587
Agent 工具循环体缩进错误 — 整个 for 循环只有一行代码

这是本次诊断发现的最严重 Bug。for round_i in range(MAX_TOOL_ROUNDS): 的循环体只包含 retry_count = 0 一行。第 588-770 行的所有工具执行代码（LLM 调用、工具分发、结果收集）都不在 for 循环内，而是与 for 同级，仅在 while 循环中执行一次。

影响：Agent 工具循环完全失效。MAX_TOOL_ROUNDS(150) 形同虚设，实际每轮 while 迭代只执行一次工具调用。最大工具轮次从 150 降为 while 的 MAX_AUTO_CONTINUES(5)。多步推理/编程任务可能无法完成。

# 当前代码（错误）： for round_i in range(MAX_TOOL_ROUNDS): retry_count = 0 # ← 循环体只有这一行！ # 使用当前动态模式... ← 这一行及以下都不在 for 循环内 from tools.dispatcher import get_mode as get_dynamic_mode current_mode = get_dynamic_mode() resp = await provider.chat_non_stream(...) # 只执行一次
修复：将第 588-770 行（从注释到 auto-continue 逻辑前）全部缩进 4 格，使它们成为 for 循环体。然后修复关联的 break/continue 控制流（见下一条）。
致命 逻辑错误 backend/api/chat.py:788-800
Auto-continue 控制流混乱 — 死代码和错误的 break 逻辑

第 795 行的 break 在 continue(793行) 之后，是死代码。第 800 行的 break 与 if/else 同级，但永远不可达（因为 else 分支已经 break，pending 分支已 continue）。

修复：重新设计控制流。在修复缩进后，确保：pending=True → continue for 循环；task 真正完成 → break for 循环；task 被截断 → break while 循环。
致命 逻辑错误 backend/tools/dispatcher.py:288
子任务模式过滤完全失效 — sub_tools 被无条件覆写

第 274-281 行按模式（ask/plan/craft）计算受限工具集，但第 288 行 sub_tools = TOOLS 无条件覆写了前面的所有过滤逻辑。子 Agent 无论什么模式都获得全部工具（包括写文件、执行命令）。

修复：删除第 288 行，或仅在 implement 模式下赋予完整 TOOLS。

附加发现

中危杂项renderer/js/gen.js
gen.js 是 Node.js 构建脚本但放在 renderer 目录中可能被 HTTP 暴露
此文件使用 require('fs') 和 require('electron')，是构建工具而非前端代码。放在 renderer/js/ 中可通过 HTTP 访问，存在信息泄露风险。
修复：移出 renderer 目录，或添加到静态文件黑名单。
中危杂项根目录 nul
根目录存在 72 字节的 nul 文件 — 命令残留
内容为 "ERROR: Invalid argument..."，是 taskkill 命令的 stderr 被错误重定向到文件 "nul" 而非 NUL 设备的产物。
修复：删除该文件，检查启动脚本中的重定向语法。
低危技术债backend/api/tools.py
api/tools.py 为空文件，工具逻辑在 chat.py 底部
路由分离不完整，工具管理端点散落在 chat.py:1101-1379 行。
低危技术债backend/api/chat.py
chat.py 文件过长 (1400+ 行)，需拆分
包含会话管理、工具管理、对话流控制、记忆注入、摘要压缩等职责，违反了单一职责原则。
低危技术债test_mcp_full.py
